# Revenue Follow-Up Flow Plan - Phase 1

Date: 2026-04-24

## Phase Boundary

Current phase: Phase 2 - live Dataverse tables and Power Apps Admin Panel MVP.

Functional after this phase:

- Resolver flow logic, quote total logic, work item generation logic, exception writing, and alert log stubs are documented.
- Flow behavior is designed to use policy and aliases rather than hardcoded people, branches, or thresholds.
- Phase 1 explicitly avoids real alert sending.
- Flow behavior is hardened to read policy operators/modes, write source system, use quote-line exception links, and count attempts from action rows.
- Planned flow behavior now preserves sticky notes and updates last followed-up / completed attempts from work item action rows.
- First follow-up is confirmed for MVP as next business day after import using `qfu_firstfollowupbasis = ImportDate` and `qfu_firstfollowupbusinessdays = 1`.

Not functional yet:

- No Power Automate flows have been built or enabled from this plan.
- No work items or assignment exceptions are generated by a live flow yet.
- No digest, targeted alert, escalation, or Teams/email delivery is active.
- No resolver flow, work item generator, or alert stub has been created in live Power Automate.
- No custom pages or Google Stitch prototypes have been created.

What comes next:

- Create child-flow scaffolding after the Phase 2 table/app foundation is reviewed.
- Run a non-alerting dry run against SP830CA quote lines.
- Review resolver output and unmapped exceptions before enabling write mode.

Still left:

- Actual flow creation, connection references, environment variables, run history validation, alert templates, dedupe validation, and replay support.

Questions that must not be guessed:

- CSSR targeted alert behavior.
- GM/manager CC trigger rules.
- Backorder and pickup flow source/grain.

## Flow Principles

- Keep Power Automate as the ingestion, normalization, and orchestration layer.
- Use child flows for reusable resolver and policy logic.
- Do not mix mailbox trigger, parsing, transformation, work item generation, alerting, and summary sync in one giant flow.
- Do not hardcode staff, emails, GMs, managers, branch IDs, thresholds, AM numbers, CSSR numbers, or source-specific exceptions.
- Do not send live alerts in Phase 1 unless explicitly enabled after review.
- Write enough trace information to validate resolver behavior before enabling alerts.
- Import/re-import must preserve work item sticky notes, action history, completed attempts, last followed-up date, status, next follow-up date, and owner fields unless a controlled reset/reassignment process is explicitly run.

## Phase 1 Flow Components

### 1. Staff Alias Normalizer

Input:

- source system
- alias type
- raw alias value

Output:

- raw alias
- normalized alias
- valid/invalid flag
- invalid reason

Rules:

- trim whitespace
- uppercase text aliases
- collapse repeated spaces
- convert Excel decimal numeric aliases such as `<staff-number>.0` to `<staff-number>`
- reject blank, `0`, `00000000`, `N/A`, `NA`, `NULL`, `NONE`

### 2. Staff Alias Resolver

Input:

- source system
- alias type
- raw alias value
- branch
- role hint
- display name from source

Output:

- resolved staff, nullable
- alias row, nullable
- assignment exception payload, if unresolved
- resolution trace

Resolution order:

1. Branch-scoped number alias.
2. Global number alias.
3. Branch-scoped verified name alias.
4. Global verified name alias.
5. Assignment exception.

Numbers must beat names.

### 3. Branch Policy Resolver

Input:

- branch
- work type

Output:

- active branch policy if present
- active global default policy otherwise
- `qfu_thresholdoperator`
- `qfu_workitemgenerationmode`
- `qfu_firstfollowupbasis`
- `qfu_firstfollowupbusinessdays`
- `qfu_gmccmode`
- `qfu_managerccmode`
- `qfu_cssralertmode`
- assignment exception payload if no policy is found

Behavior:

- Branch policy overrides global default.
- Missing policy creates `qfu_assignmentexception` with `qfu_exceptiontype = Missing Policy`.
- Flow must not use hardcoded threshold or attempts when policy lookup fails.
- Flow must not infer GM, manager, or CSSR alert behavior from enabled-only booleans when mode fields exist.
- Duplicate active policies for the same scope/work type are an open enforcement decision; flows should fail visible rather than picking an arbitrary duplicate.

### 4. Quote Total Calculator

Input:

- branch
- quote number
- quote line rows

Output:

- quote total
- source quote/header lookup, if available
- source line count
- source external key

Rule:

```text
Quote total = sum of Value for all active/current lines with same branch + quote number
```

Confirmed live source fields:

- `qfu_quoteline.qfu_tsr` = AM Number / TSR numeric alias.
- `qfu_quoteline.qfu_tsrname` = AM Name / TSR display name.
- `qfu_quoteline.qfu_cssr` = CSSR numeric alias.
- `qfu_quoteline.qfu_cssrname` = CSSR display name.
- `qfu_quoteline.qfu_amount` = quote line value.

`qfu_quote` can be linked as the quote header/current-state row, but `qfu_quoteline` should drive identity resolution and quote total calculation in Phase 1 because it has the full numeric and display identity set.

### 5. Quote Work Item Generator

Input:

- branch
- quote number
- grouped quote lines
- policy
- TSR resolver output
- CSSR resolver output

Output:

- create/update `qfu_workitem`
- create/update required `qfu_assignmentexception` rows

Core behavior:

1. Import quote data into `qfu_quote` and `qfu_quoteline` as today.
2. Group quote lines by branch and quote number.
3. Calculate quote total.
4. Resolve policy.
5. Use `qfu_policy.qfu_thresholdoperator` and `qfu_policy.qfu_workitemgenerationmode` to decide whether to create/update `qfu_workitem`.
6. Normalize `qfu_tsr` and `qfu_cssr`.
7. Resolve `qfu_tsr` through `qfu_staffalias` as TSR.
8. Resolve `qfu_cssr` through `qfu_staffalias` as CSSR.
9. If `qfu_tsr` is blank, zero, invalid, unmapped, or ambiguous, set assignment status to `Needs TSR Assignment` and create an exception.
10. If `qfu_cssr` is blank, zero, invalid, unmapped, or ambiguous, create an exception but allow TSR-owned work item when TSR is resolved.
11. Store required attempts from policy.
12. Calculate first follow-up from `qfu_firstfollowupbasis` and `qfu_firstfollowupbusinessdays`; Phase 2 MVP defaults are `ImportDate` and `1`.
13. Write `qfu_sourcesystem = SP830CA` on quote work items.
14. Write a representative `qfu_sourcequoteline` on assignment exceptions when useful.
15. Preserve any existing sticky note, action-history-derived fields, status, next follow-up, and owner overrides on re-import.
16. Do not send alerts in Phase 2.

Generation mode behavior:

| Mode | Behavior |
| --- | --- |
| HighValueOnly | Create/update work items only when quote total satisfies the configured threshold operator. This is the MVP quote work item creation mode; below-threshold quotes remain reporting-only. |
| AllQuotes | Create/update quote work items regardless of threshold; still store total value and policy. |
| ReportingOnly | Do not create workflow work items; allow reporting/dry-run counts only. |

Threshold operator behavior:

| Operator | Comparison |
| --- | --- |
| GreaterThan | `quoteTotal > qfu_highvaluethreshold` |
| GreaterThanOrEqual | `quoteTotal >= qfu_highvaluethreshold`; this is confirmed for the MVP `$3,000` quote threshold. |

First follow-up behavior:

| Basis | Behavior |
| --- | --- |
| ImportDate | Add `qfu_firstfollowupbusinessdays` to the import date. |
| QuoteDate | Add `qfu_firstfollowupbusinessdays` to the source quote date. |
| SourceDueDate | Use the source due date when present; create an exception if required and missing. |
| Manual | Leave `qfu_nextfollowupon` blank for admin/staff assignment. |
| Disabled | Do not set an initial follow-up date. |

Confirmed MVP default:

```text
qfu_firstfollowupbasis = ImportDate
qfu_firstfollowupbusinessdays = 1
Meaning: next business day after import for high-value quote work items.
```

Business-day timezone and branch holiday handling are later flow concerns. Do not invent branch holiday rules in Phase 2.

Upsert key:

```text
qfu_worktype + qfu_sourceexternalkey
```

Quote source external key:

```text
branch|SP830CA|quote|<quote-number>
```

### 6. Assignment Exception Writer

Input:

- exception type
- branch
- source system
- source field
- raw value
- normalized value
- display name
- source document number
- source external key
- source record lookup
- optional source quote line lookup
- work item lookup

Output:

- create/update `qfu_assignmentexception`

Recommended idempotency:

```text
qfu_exceptionkey = sourcesystem|sourcefield|normalizedvalue|sourceexternalkey|scopekey
```

If the implementation cannot add `qfu_exceptionkey` in the first table build, the flow must still query for matching open exceptions before creating a new one.

For quote-line-driven resolver failures, set `qfu_sourcequoteline` to a representative source line when it helps an admin trace the exception. Do not require one exception per line when the work item grain is branch + quote number.

### 7. Work Item Attempt Counter

Input:

- work item id
- related `qfu_workitemaction` rows

Output:

- updated `qfu_workitem.qfu_completedattempts`
- updated `qfu_workitem.qfu_lastfollowedupon`
- updated `qfu_workitem.qfu_lastactionon`

Behavior:

- Count only actions where `qfu_countsasattempt = Yes`.
- Set `qfu_workitem.qfu_completedattempts` to the count of related actions where `qfu_countsasattempt = Yes`.
- Set `qfu_workitem.qfu_lastfollowedupon` to the max `qfu_actionon` across related actions where `qfu_countsasattempt = Yes`.
- Set `qfu_workitem.qfu_lastactionon` to the max `qfu_actionon` across all related actions.
- `qfu_actionon` is the actual date/time the action happened; it should default to now for new manual actions but remain editable for earlier follow-up logging.
- Any action where `qfu_countsasattempt = Yes` must require `qfu_actionon`.

MVP default attempt-bearing actions:

| Action Type | Default `qfu_countsasattempt` |
| --- | --- |
| Call | Yes |
| Email | Yes |
| Customer Advised | Yes |
| Note | No |
| Roadblock | No |
| Escalated | No |
| Due Date Updated | No |
| Won | No |
| Lost | No |
| Cancelled | No |
| Assignment/Reassignment | No |
| Sticky Note Updated | No |

Admins can change defaults later through policy or configuration if needed.

### 8. Sticky Note Updater

Input:

- work item id
- sticky note text
- updated by staff, nullable
- update timestamp

Output:

- updated `qfu_workitem.qfu_stickynote`
- updated `qfu_workitem.qfu_stickynoteupdatedon`
- updated `qfu_workitem.qfu_stickynoteupdatedby`
- optional `qfu_workitemaction` audit row with `qfu_actiontype = Sticky Note Updated` and `qfu_countsasattempt = No`

Behavior:

- Sticky note updates should never write to imported `qfu_quote`.
- Sticky note updates should stay with the work item even when source SP830CA rows refresh.
- Import/re-import should never overwrite sticky notes or action history.

### 9. Alert Log Stub

Phase 1 should design but not send alerts.

When alerting is enabled later, every alert must write `qfu_alertlog` before or during send with a stable dedupe key.

Recipient behavior must come from policy mode fields:

| Policy field | Controls |
| --- | --- |
| `qfu_gmccmode` | When GM recipients are copied or included in digest/escalation output. |
| `qfu_managerccmode` | When manager recipients are copied or included in digest/escalation output. |
| `qfu_cssralertmode` | Whether CSSR is visibility-only, digest-only, targeted, CC-only, or disabled. |

Do not send live alerts in Phase 1.2; only validate that policy output can be resolved without hardcoded people or emails.

Candidate dedupe keys:

| Alert Type | Dedupe Key Shape |
| --- | --- |
| New Assignment | `workitem|new-assignment|recipient-staff` |
| Due Today | `workitem|due-today|recipient-staff|yyyy-mm-dd` |
| Overdue | `workitem|overdue|recipient-staff|yyyy-mm-dd` |
| Daily Digest | `digest|branch|recipient-staff|yyyy-mm-dd` |
| Escalation | `workitem|escalation-level|recipient-staff` |

## Dry Run Validation

Before enabling writes:

- Run resolver against recent SP830CA rows without creating alerts.
- Write summary counts to `results/`.
- Capture:
  - rows scanned
  - quote groups scanned
  - high-value quote groups
  - TSR aliases resolved from `qfu_tsr`
  - CSSR aliases resolved from `qfu_cssr`
  - blank/zero `qfu_tsr` exceptions
  - blank/zero `qfu_cssr` exceptions
  - unmapped aliases
  - ambiguous aliases
  - missing policies
  - would-create work items
  - would-update work items

Before enabling alert sending:

- Confirm Admin Panel policy rows exist.
- Confirm active staff have emails where alerts are expected.
- Confirm branch memberships include Manager/GM where CC or escalation policies require them.
- Confirm `qfu_alertlog` dedupe keys suppress duplicate sends.

## Error Handling

- Resolver failures should create `qfu_assignmentexception` or a flow-failure alert log stub, not silently skip rows.
- Missing branch or missing policy should be visible to admins.
- Flow run id should be captured on failure logs where possible.
- Retry logic should be idempotent and should not create duplicate work items, exceptions, or alert logs.

## Phase 1 Tests

Recommended test cases:

- `<staff-number>` normalizes to `<staff-number>`.
- `<staff-number>.0` normalizes to `<staff-number>`.
- blank AM Number creates `Blank Alias` or `Missing TSR Alias` exception.
- `0` AM Number creates `Zero Alias` exception.
- `00000000` CSSR creates `Zero Alias` exception.
- branch-scoped alias wins over global alias.
- number alias wins over name alias.
- unmapped AM Number creates `Needs TSR Assignment`.
- unmapped CSSR creates CSSR exception but does not block resolved TSR ownership.
- quote lines aggregate to quote total before threshold comparison.
- quote total eligibility uses `qfu_thresholdoperator` and `qfu_workitemgenerationmode`, not a hardcoded comparison.
- quote total at `$3,000` qualifies when threshold operator is GreaterThanOrEqual.
- low-value quote groups remain reporting-only in the MVP.
- action with `qfu_countsasattempt = Yes` updates completed attempts and last followed-up date.
- action with `qfu_countsasattempt = No` can update last action but not last followed-up date.
- sticky note survives quote re-import.
- duplicate import updates existing work item.
- duplicate exception input updates existing exception.
