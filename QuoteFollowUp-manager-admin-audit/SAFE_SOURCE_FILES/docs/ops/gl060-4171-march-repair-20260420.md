### Root Cause

Current evidence does not support a parser defect for March `4171`. The supplied March `4171` PDF parses correctly and writes the expected March finance numbers when queued manually.

The live defect was upstream of parsing:

- the exact March `4171` PDF content never appeared in live `qfu_rawdocument`
- the live duplicate chain only referenced the old February `4171` content

That means the original April 10 `4171` March GL060 did not enter the live raw queue that the downstream processor consumes.

### Mailbox Trigger Behavior

- The Southern Alberta shared-mailbox ingress flows were normalized on April 14, 2026 and the repair artifacts show all three GL060 flows running with:
  - folder = `Inbox`
  - recurrence = `1 Minute`
  - trigger = shared mailbox new email
- This means the April 10 `4171` miss was not proven to be caused by a `10 minute` polling interval.
- The current Office 365 shared-mailbox trigger pattern is still a polling pattern, not a true push-on-arrival subscription. In other words:
  - the supported live shape is `Inbox` plus `1 minute` polling
  - a true event-driven mailbox trigger would require a different ingress design than the current shared-mailbox connector pattern
- The production reliability gap is therefore not "GL060 only checks every 10 minutes." The real gap is that a mailbox event can be missed before it creates the expected `qfu_rawdocument` audit trail.

### Durable Fix Path

- Diagnose why the original April 10 `4171` GL060 email did not land in `qfu_rawdocument`.
- Use mailbox trigger evidence and run history to prove whether the miss was:
  - shared mailbox trigger miss
  - routing/folder issue
  - message delivery issue before flow ingress
- Do not restore the retired local GL060 scheduled task as the production fix.
- Keep the current queue replay plus queue processor path for narrow incident recovery only.
- The production path still needs a proven hosted downstream GL060 consumer and a proven mailbox-ingress audit trail so the same gap cannot recur silently.
- If the business requires true arrival-based execution with no polling interval, plan a separate ingress redesign instead of expecting the current shared-mailbox trigger to behave like a push subscription.
